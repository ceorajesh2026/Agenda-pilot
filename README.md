# AgendaPilot — WNS 2026 Smart Agenda Manager

Build covering **Milestones M0, M1, M2 and M3** of the [implementation plan](AgendaPilot-Implementation-Plan.md).
It ingests the real WNS 2026 source files into a normalized schema, links program speakers to the
faculty directory, runs a deterministic conflict engine, renders an organizer dashboard, and drives the
full disruption-recovery loop (report → AI options → two-step approval → publish → notify → acknowledge).

> **AI proposes, rules verify, humans approve.** Option *generation* is deterministic constraint-solving;
> every candidate is validated by the M2 engine before a human sees it; nothing publishes without both
> human gates. The natural-language rationale + notification copy are generated by **Claude** (`claude-opus-4-8`
> via a small server-side proxy), with deterministic templates as an offline fallback — see *The prose seam* below.

## Architecture

```
source files (.xlsx/.docx)
        │
        ▼   etl/  (Python, one-time import — stands in for the future in-app import wizard)
  build_seed.py ──► app/src/data/seed.json   +   etl/build/resolution-review.csv
        │
        ▼   app/  (Vite + React + TypeScript)
  lib/constraints.ts  ── deterministic engine (the "rules verify" gate)
  App.tsx             ── organizer dashboard (6 views)
```

**Stack (kickoff):** Vite + React + TypeScript, plain CSS control-room theme. The heavy, stack-neutral
work — schema, parsing, entity resolution, constraint rules — is isolated so it ports directly to the
production target (**Next.js on Vercel + Supabase Postgres**, per the plan). `seed.json` becomes Postgres
tables; `constraints.ts` stays as-is; the ETL becomes the in-app import wizard.

## How to run

```bash
# 1) (re)build the data from the source files  — needs: pip install openpyxl python-docx
cd etl && python build_seed.py

# 2) run the dashboard
cd app && npm install && npm run dev      # http://localhost:5173
npm run build                             # typecheck + production build

# 3) run the server — powers M4 distribution (feeds/ICS/print/publish) AND the M3 Claude prose.
#    The API key is only needed for Claude-drafted copy; distribution works without it.
cd app && ANTHROPIC_API_KEY=sk-ant-... npm run server   # http://localhost:8787
#    Vite proxies /api -> :8787; the key never reaches the browser. See .env.example.
```

## Data model (normalized in `seed.json`)

`Event → Track → Day → Hall → Session → Slot(kind) → Role(role_type) → Person`, plus entity-resolution
metadata. Mirrors the PRD's core entities. Halls are **provisional** (no room data in the source —
Open Question #6); neurosurgery sessions are **DRAFT** (no times/speakers in the source).

## Key modules

- [`etl/build_seed.py`](etl/build_seed.py) — parses the Neurology program (time normalization, free-text
  role parsing, umbrella-workshop nesting) + both neurosurgery Word docs; runs fuzzy entity resolution.
- [`etl/directory.py`](etl/directory.py) — cleans/dedupes the two faculty spreadsheets into 857 Person records.
- [`app/src/lib/constraints.ts`](app/src/lib/constraints.ts) — the deterministic checker: person double-booking,
  hall clash, missing moderator, duration drift (±10 min), unlinked speaker, unreachable faculty, DRAFT backlog.
- [`app/src/App.tsx`](app/src/App.tsx) — the six dashboard views.

## The M3 loop (Disruption Agent tab)

Runnable end-to-end against the flagship **Journey B** scenario (built-in preset):
1. Dr. Divyanshu Dubey reports a flight delay (ETA 13:30, Day 2).
2. The agent maps the impact set and generates **3 options**, each validated by the M2 constraint engine
   and scored (lower = less disruptive): **Swap with Neuro-Ophthalmology 1 (recommended, 18)** ·
   Substitute M V Padma Srivastava — a real autoimmune-neurology backup (25) · Absorb/compress (29).
3. Two human gates: **Chair** (Approve / Approve-with-edits / Reject) → **Secretariat** (Confirm & Publish / Send back).
4. Publish v2 → **4 targeted notifications** (email primary, WhatsApp where numbers exist) with Acknowledge buttons.
5. **Rollback to v1** + a timestamped **audit trail** of every actor and action.

> A worked example of "rules verify": the naive swap that overflowed a 35-min session into the next
> slot was **rejected by the constraint checker**, so the agent only surfaces the window-swap that fits.

## The prose seam — wired to Claude

Option *generation* is deterministic and constraint-validated; the natural-language **rationale +
notification copy** are the LLM's job. That seam is now wired:

- [`app/server/index.mjs`](app/server/index.mjs) — a tiny Node/Express proxy that holds `ANTHROPIC_API_KEY`
  and calls **`claude-opus-4-8`** via the official `@anthropic-ai/sdk`, using **strict tool use + forced
  `tool_choice`** so the returned JSON (`{rationale, notifications[]}`) is schema-guaranteed. It is sent
  the *already-validated* plan (what swaps, who's affected, the times) and only writes the copy — it never
  invents a plan.
- [`app/src/lib/prose.ts`](app/src/lib/prose.ts) — the browser client. On any failure (no server, no key,
  network down) it returns `null` and the UI falls back to the deterministic templates — so the recovery
  loop never depends on the network (PRD offline-tolerance).
- A **"✨ Draft copy with Claude"** toggle in the Disruption Agent enables it; a Claude-drafted option shows
  a badge. Without the server the toggle is disabled and the app uses templates.

## M4 — Distribution (📡 Distribution tab)

On Secretariat approval the app POSTs the revision to the server, which applies it to the published agenda
(seed + approved revisions) and propagates it — verified to land in **all channels within the same second**:

- **Website** — `GET /api/public/agenda.json` (feed), `GET /api/public/widget.js` (one `<script>` tag,
  platform-agnostic per PRD Q1), `GET /api/public/embed` (iframe). Revised sessions carry a "REVISED" badge
  and a "last updated" stamp; all are `Access-Control-Allow-Origin: *` for cross-site embedding.
- **Calendars** — `GET /api/ics/agenda.ics` (whole event) and `GET /api/ics/person/<id>.ics` (auto-updating
  per-person feed) with `SEQUENCE` bumps so subscribers see the change.
- **Print** — `GET /api/print/day/<date>` renders a print-ready day program (Ctrl/Cmd-P → Save as PDF) for
  the registration desk.
- **WhatsApp groups** — a formatted change-summary post (Faculty + Delegates).
- **Reliability** — replaces the PRD's fragile "atomic" publish: each target has its own status, notifications
  are **rate-limited + idempotent** with a `deliver()` seam for real Meta/Twilio/SES, and a **change +
  notification log** (`GET /api/changelog`) records every publication. If the server is down, the revision
  still applies locally (offline-tolerant).

## What's intentionally NOT here yet (beyond M4)

- **Real** message *delivery* — the `deliver()` adapter records `sent`/`needs_call`/`rate_limited` statuses
  but doesn't yet call Meta/Twilio/SES; print is print-to-PDF (a `pdfkit` renderer is a drop-in).
- Escalation timers as live countdowns (shown as static SLA chips today).
- Auth/RBAC, personal-link self-report + collection form — production migration.
- Persistence: workflow/publication state is in-memory (resets on server restart) — moves to Postgres/Supabase.
